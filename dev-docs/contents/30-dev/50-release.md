# リリースフロー

[Q6](node:/tasks/Q/tasks/Q6)（手順の明文化）/ [Q10](node:/tasks/Q/tasks/Q10)（ワークフロー実装）の検討メモ。
実装完了時にチェックリスト本体へ置き換える。

## 決定事項（大枠）

- **main は trunk のまま、リリースは別イベント**。「main マージ = リリース」にはしない。
  リリースは version 確定 + git tag push であり、tag push を契機に GH Actions が
  build → PyPI publish（trusted publishing）→ GH Release 作成を行う
- **バージョニング**: 0.x 慣習に従い 0.MINOR を破壊的変更に予約、それ以外は 0.x.PATCH。
  破壊的変更 ⇔ フォーマット世代 +1 の 1:1 対応（[sbdb-manifest](node:/prose/20-design/20-app/60-sbdb-manifest)
  の世代台帳が判定の照合先）。1.0.0 はローンチ（[Q3](node:/tasks/Q/tasks/Q3)）で刻む
- **リリースの契機は裁量**（少なくともローンチまで）。契機は版種別で二層に割れる:
  patch は世代判定・移行手順が不要で原理的に安く、任意のタイミングで打てる。
  minor は「破壊的変更を出す」という決断そのものが契機であり、自動化する余地がない
- **ブランチモデルは trunk-based 維持、Git Flow は採らない**。Git Flow は選択的取り込み
  （破壊的変更を除いてリリースを切る）を構造では解けず、常設ブランチの維持コストだけが残る。
  旧系列へのパッチ需要（= 新 minor に上がれない利用者）が実際に発生したら、
  過去 tag から遡及ブランチを切って cherry-pick する（`git switch -c release-0.X v0.X.Y`）。
  需要はローンチ前にはほぼ定義上存在しない
- **破壊的 PR のマージタイミング**: 破壊的 PR をマージすると、以後の main からは
  patch リリースが打てなくなる（minor を仕上げて出すか、遡及ブランチに頼る）。
  「次のリリースを minor にする覚悟ができたときにマージする」を習慣とする。
  規律が払えないコストになったときの構造的解は Git Flow ではなく、
  sbdb_version 対応集合の複数世代並行サポート（sbdb-manifest に YAGNI として保留済み）

## チェックリスト骨格（ドラフト）

前段の判定ブロック → 版種別の経路、という構造。

```
■ 前段（毎回共通）
1. git log <前回タグ>..HEAD で変更を棚卸し
2. 破壊的変更の有無を判定 → 無ければ patch 経路 / 有れば minor 経路

■ patch 経路（5 分で回る軽さを目標）
3. version 確定 → showcase / dev-docs の minimum_version 更新
4. tag push → ワークフロー完走・PyPI 反映の確認

■ minor 経路（patch 経路 + 以下）
3'. sbdb_version +1: コードの対応集合更新 + 世代台帳への追記
    （破壊的 PR 自身が済ませているはず — 下記フラグ管理）
4'. 移行手順をリリースノート冒頭に手書き
5'. scaffold 生成物の sbdb_version 追随
```

## 破壊的変更フラグの管理

「前回リリース以降に破壊的変更がマージされたか」をリリース時にどう知るか。
判定を PR マージ時点（文脈が最も新鮮）に前倒しし、リリース時は機械的照会に潰すのが方針。

- **世代番号 in-tree**: 破壊的 PR は定義上フォーマット世代を上げる PR であり、
  対応集合・世代台帳の書き換えを自身に含む。よって「コード内の世代番号が前回 tag から
  変わったか」の diff 一発で判定できる。フラグを立てる行為と破壊的変更が同一物なので
  貼り忘れが構造的に不可能
- **隙間**: フォーマットを変えないが利用者を壊す変更（CLI フラグ廃止等）は世代が動かず
  ツリーに現れない。PR ラベル（`breaking`）を保険で併用するか、「それも世代 +1 とみなす」
  と規約を広げるかは未決（下記残論点）
- versioning 方式 C を採る場合、pyproject の version 宣言そのものがフラグを兼ねる（次節）

## versioning 方式（選定待ち）

三択を検討し、実質二強に絞った。相互排他（C は動的導出と両立しない）。

| | 方式 | フラグの所在 | リリース時の版作業 |
|---|---|---|---|
| A | 静的 + リリース時 bump | ラベル or 世代番号 diff | bump コミット |
| B | hatch-vcs（git tag から動的導出） | 世代番号 diff | なし（tag のみ） |
| C | 静的 + 常時 `.devN` 仮宣言 | **pyproject そのもの** | `.dev0` 剥がし |

- **B**: 版の宣言が tag ただ一つになり、bump というステップ自体が消滅。
  tag/pyproject 不一致というエラークラスごと消える。開発中ビルドは
  `0.1.2.dev3+g1a2b3c4` 形式で自動的に正直。トレードオフ: ビルド時に git メタデータが
  必要、hatch-vcs というビルド依存の追加、版番号がどこにも書かれないことへの慣れ
- **C（完成形）**: Maven の SNAPSHOT 運用の PEP 440 移植。リリース直後に次 patch の
  仮宣言 `0.4.1.dev0` を立て、破壊的 PR が `0.5.0.dev0` に上書きする（これがフラグ）。
  リリース時の版作業は版種別によらず「`.dev0` 剥がし」の一種類。仮宣言が外れた番号は
  欠番になるだけで無害。トレードオフ: リリース直後の仮宣言立てという機械的コミットが
  毎リリース +1（チェックリスト 1 行 or ワークフロー後処理で自動化可）
- A は C の下位互換（常時仮宣言がないぶん main の版番号が嘘をつく）なので落選

選定はチェックリスト明文化（Q6）の中で行い、選定理由をここに記録する。

## 自動化の実装（Q10 / Q11）

自動化は契機の異なる二面に分かれる。tag push 側は publish の実行、
PR マージ側は規律のゲート化であり、性格が違う。

### tag push 側 — publish 実行（[Q10](node:/tasks/Q/tasks/Q10)）

- tag push（`v*`）契機: build → PyPI publish → GH Release 作成
- 認証は trusted publishing（PyPI 側にリポジトリ + ワークフローファイル名を登録。
  プロジェクトは 0.1.0 手動 publish（[Q1](node:/tasks/Q/tasks/Q1)）で実在するため通常登録。
  以後 API token は不要になる）
- tag と pyproject の version 一致検証（方式 B を採る場合は不要）
- リリースノートは GH 自動生成（PR タイトルの束ね）。`.github/release.yml` のラベル別節で
  breaking を分離し、破壊的リリースのみ冒頭に移行手順を手書きする。
  外部向けの体裁格上げ判断は [Q7](node:/tasks/Q/tasks/Q7)

### PR マージ側 — 破壊的変更ゲート（[Q11](node:/tasks/Q/tasks/Q11)）

破壊的変更が「しれっと」マージされるのを PR 時点で捕まえる追加プロセス。
何かを置くことはほぼ確定、実装形が未確定。候補（併用可）:

- **ci.yml への検査 job**（機械化できる規律）: フラグ整合（世代番号を触る PR ⇔
  breaking ラベル）、方式 C の `.devN` 規律検査
- **AI レビュー**: merge 前に「破壊的変更が紛れていないか」を観点指定でチェックさせる
- **`.github/pull_request_template.md`**: 破壊的変更の有無を自己申告するチェック項目
- **DEVELOPMENT.md の規律明文化**: 人間の習慣として残す部分の文書化

minimum_version 整合（showcase / dev-docs）は GH Actions ではなく通常の
テストスイートに書ける可能性がある。どの手段をどの深さで採るかの捌きは
[Q6](node:/tasks/Q/tasks/Q6) で決める。実装先は新規ワークフローではなく
既存 ci.yml への追加を想定。
